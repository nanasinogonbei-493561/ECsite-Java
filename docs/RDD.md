# 日本酒ECサイト要件定義書（1人開発版）

## 1. プロジェクト概要

### 1.1 目的
日本酒のオンライン販売サイトをMVP（最小実用可能製品）として構築し、基本的な販売機能を提供する。

### 1.2 開発制約
- **開発人数**: 1人
- **開発期間**: 1ヶ月（実働20日）
- **開発言語**: バックエンド: Java + Spring Boot
              フロントエンド: React + TypeScript
- **スコープ**: 必要最小限の機能に特化

### 1.3 取り扱い商品
- 日本酒（720ml、1800mlボトル）
- 初期商品数: 20-30商品程度

## 2. 機能要件（超シンプル版）

### 2.1 顧客向け機能

#### 2.1.1 商品機能（最小限）
- **トップページ**
  - 商品一覧（シンプルなカード表示）
  - 簡単な検索（商品名のみ）
- **商品詳細ページ**
  - 商品名、価格、容量
  - 商品画像（1枚）
  - 醸造元名
  - アルコール度数
  - 簡単な商品説明
  - 在庫状況（在庫あり/なし）

#### 2.1.2 年齢確認（必須）
- **初回アクセス時の年齢確認**
  - 生年月日入力
  - 20歳未満はアクセス拒否
  - セッション管理（ブラウザ閉じるまで有効）

#### 2.1.3 注文機能（シンプル）
- **注文フォーム**
  - 商品詳細ページから直接注文
  - 顧客情報入力（名前、住所、電話、メール）
  - 数量選択（1-3本まで）
- **注文確認画面**
  - 注文内容確認
  - 合計金額表示
- **注文完了画面**
  - 注文番号表示
  - 確認メール送信

#### 2.1.4 決済
- **代金引換のみ**
  - クレジットカード決済は実装しない
  - 銀行振込も後回し
  - 配送業者による代金回収

### 2.2 管理者向け機能（最小限）

#### 2.2.1 商品管理
- **商品一覧画面**
  - 登録商品の一覧表示
- **商品登録画面**
  - 基本項目のみ（名前、価格、醸造元、アルコール度数、在庫数）
  - 画像アップロード（1枚のみ）
- **商品編集・削除**

#### 2.2.2 注文管理
- **注文一覧画面**
  - 受注一覧
  - ステータス管理（受注→発送済み→完了）
- **注文詳細画面**
  - 顧客情報、注文内容確認

#### 2.2.3 管理者ログイン
- **シンプルな認証**
  - 管理者ID/パスワードによるログイン
  - セッション管理

## 3. 技術要件（シンプル構成）

### 3.1 技術スタック
- **バックエンド**: Java 17 + Spring Boot 3.0
- **フロントエンド**: React19.1.1+TypeScript5.8.3
- **データベース**: 
  - 開発環境: H2 Database
  - 本番環境: PostgreSQL（Render）またはMySQL（ConoHa VPS）
- **ビルド**: Maven
- **デプロイ**: 
  - **選択肢1**: ConoHa VPS + Docker
  - **選択肢2**: Render（PaaS）

### 3.2 アーキテクチャ（シンプル構成）
```
Controller → Service → Repository → Database
```

### 3.3 デプロイ環境詳細

#### 選択肢1: ConoHa VPS
- **サーバー**: ConoHa VPS（1GB RAM推奨）
- **OS**: Ubuntu 22.04 LTS
- **コンテナ**: Docker + Docker Compose
- **データベース**: MySQL 8.0（Docker）
- **Webサーバー**: Nginx（リバースプロキシ）
- **SSL**: Let's Encrypt
- **ファイル保存**: VPS内ローカルストレージ

#### 選択肢2: Render
- **プラットフォーム**: Render（PaaS）
- **データベース**: PostgreSQL（Render提供）
- **ファイル保存**: Cloudinary（画像）
- **SSL**: 自動提供
- **スケーリング**: 自動対応
- **料金**: $7/月〜（スタータープラン）

### 3.4 主要エンティティ（最小限）
1. **Product（商品）** - 商品情報
2. **Order（注文）** - 注文ヘッダー
3. **OrderItem（注文明細）** - 注文商品詳細
4. **Admin（管理者）** - 管理者情報

### 3.5 環境変数設定
```properties
# 本番環境用
SPRING_PROFILES_ACTIVE=prod
DATABASE_URL=（接続文字列）
DATABASE_USERNAME=（DBユーザー名）
DATABASE_PASSWORD=（DBパスワード）
CLOUDINARY_URL=（画像アップロード用、Renderの場合）
MAIL_PASSWORD=（メール送信用）
```

## 4. データベース設計（シンプル版）

### 4.1 商品テーブル (products)
```sql
- id (BIGINT, PK, AUTO_INCREMENT)
- name (VARCHAR(255)) - 商品名
- brewery (VARCHAR(100)) - 醸造元
- price (DECIMAL(10,2)) - 価格
- volume (INTEGER) - 容量(ml)
- alcohol_content (DECIMAL(3,1)) - アルコール度数
- description (TEXT) - 説明
- image_path (VARCHAR(255)) - 画像パス
- stock_quantity (INTEGER) - 在庫数
- created_at (DATETIME)
- updated_at (DATETIME)
```

### 4.2 注文テーブル (orders)
```sql
- id (BIGINT, PK, AUTO_INCREMENT)
- order_number (VARCHAR(20), UNIQUE) - 注文番号
- customer_name (VARCHAR(100)) - 顧客名
- customer_email (VARCHAR(255)) - メールアドレス
- customer_phone (VARCHAR(20)) - 電話番号
- delivery_address (TEXT) - 配送先住所
- total_amount (DECIMAL(10,2)) - 合計金額
- status (VARCHAR(20)) - ステータス
- created_at (DATETIME)
```

### 4.3 注文明細テーブル (order_items)
```sql
- id (BIGINT, PK, AUTO_INCREMENT)
- order_id (BIGINT, FK)
- product_id (BIGINT, FK)
- quantity (INTEGER) - 数量
- unit_price (DECIMAL(10,2)) - 単価
```

### 4.4 管理者テーブル (admins)
```sql
- id (BIGINT, PK, AUTO_INCREMENT)
- username (VARCHAR(50), UNIQUE)
- password (VARCHAR(255)) - ハッシュ化
```

## 7. 実装の優先順位

### 優先度：高（必須実装）
- 年齢確認機能
- 商品表示機能
- 注文機能（代金引換）
- 管理者による商品・注文管理

### 優先度：中（時間があれば）
- 簡単な検索機能
- 注文確認メール
- 在庫管理

### 優先度：低（Phase2対応）
- 会員機能
- 複数決済手段
- レビュー機能
- 高度な検索・フィルタ

## 8. 制約事項・妥協点

### 8.1 機能面での妥協
- 会員登録機能なし（都度入力）
- 決済は代金引換のみ
- ショッピングカート機能なし
- レコメンド機能なし
- 詳細な商品情報（日本酒度等）は省略

### 8.2 技術面での妥協
- セキュリティは基本的なもののみ
- ログ機能は最小限
- パフォーマンス最適化は後回し
- 本番データベースはシンプル構成

### 8.3 運用面での妥協
- 自動バックアップは手動設定
- 監視機能は最小限
- 負荷分散なし（単一インスタンス）

### 8.4 デプロイ環境の選択基準
- **ConoHa VPS**: サーバー管理に慣れている場合、コスト重視
- **Render**: 手軽さ重視、サーバー管理不要、GitHubからの自動デプロイ

## 9. コスト概算

- VPS 1GB: 月額696円〜
- ドメイン: 年額1,000円程度
- 合計: 月額約800円

## 10. リスク・課題

### 10.1 主要リスク
- 1人開発による開発遅延
- 酒類販売免許の法的要件対応
- デプロイ環境の選択による影響
- 本番環境でのトラブル対応

### 10.2 対応策
- 機能を最小限に絞る
- 外部ライブラリを積極活用
- 段階的リリース（機能追加は後日）
- デプロイ環境は経験・予算に応じて選択

### 10.3 デプロイ環境別リスク

#### ConoHa VPS
- **リスク**: サーバー管理の複雑さ、セキュリティ設定
- **対策**: Docker使用で環境統一、基本的なセキュリティ設定

#### Render
- **リスク**: 料金の高さ、プラットフォーム依存
- **対策**: 初期はRenderで開始、後にVPS移行も検討

- PostgreSQL簡単セットアップ

### 経験者・コスト重視 → **ConoHa VPS**
